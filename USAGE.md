# Best Practices and Usage Guide

Welcome to the definitive guide for using the Bug Bounty Prompt Library. This documentation will teach you how to properly utilize the **System Harness** and the **Pluggable Skill Modules** to execute advanced security research workflows with maximum efficiency.

## 1. The Core Architecture: Harness + Skill

This repository abandons the concept of "mega-prompts" in favor of a modular, two-part architecture. 

* **The System Harness**: This is the immutable foundation. It sets the overarching persona, enforces the Chain of Thought, and establishes the strict operational constraints (e.g., zero hallucination, XML formatting).
* **The Skill Module**: These are the specific, atomic instructions that define *how* to execute a specific task, such as finding a Boolean-Based Blind SQLi or forging a JWT.

## 2. Step-by-Step Execution Workflow

To get the absolute best results from your LLM (such as Claude 3.5 Sonnet or Opus), you must sequence your interaction correctly.

### Step 1: Prime the Engine (Load the Harness)
Start a fresh chat or API session. **Do not put any target data in the first message.**
Instead, copy the contents of `system_harness.md` (or the output generated by the `harness_engineer_meta_prompt.md`) and paste it as the *System Prompt*. If you are using a standard chat interface, paste it as your very first message.

Wait for the LLM to acknowledge the constraints.

### Step 2: Select the Skill Module
Identify the specific vulnerability you are hunting for. Browse through the collections (e.g., `sqli_prompts.md`, `client_side_prompts.md`) and copy the exact "Skill Module" block that matches your target architecture.

### Step 3: Inject the Target Data
The Skill Module will have an `<input_data>` tag or a bracketed section labeled `[Insert the relevant HTTP request...]`. 
Replace this placeholder with the raw data from your proxy (e.g., Burp Suite HTTP requests, WebSocket traffic, minified JavaScript files, or JWTs).

### Step 4: Execute the Prompt
Send the combined Skill Module and your raw input data to the LLM. 

### Step 5: Analyze the Output
Because the System Harness enforces Chain of Thought, the LLM will first output its step-by-step reasoning (often within `<thinking>` or `<scratchpad>` tags). Read this section carefully. It reveals exactly how the model interpreted the input data and formulated its attack plan. 

After the thinking phase, the LLM will provide the weaponized proof-of-concept (e.g., a fully formatted `curl` command or Burp Suite HTTP payload) ready for execution.

## 3. Best Practices for High-Impact Bounties

* **Keep Context Clean**: Never reuse a chat session for different targets or different vulnerability classes. Context pollution will degrade the model's analytical rigor. Always start a fresh session, reload the Harness, and inject the new Skill Module.
* **Provide Complete Raw Data**: Do not summarize the HTTP request yourself. Paste the entire raw request and response headers. The LLM's value lies in spotting the obscure discrepancies (like a missing `X-Content-Type-Options` header) that you might have overlooked.
* **Verify Execution**: The payloads generated are theoretical. It is your job as the researcher to execute the payload in Burp Suite, observe the behavioral delta, and confirm the vulnerability. Use the output as a highly educated starting point, not the definitive end.

By adhering to this modular, disciplined approach, you transform the LLM from a generic chatbot into an elite, automated security architect. Happy hunting!
